# Pattern 2 – Q7（Task 1.7）MPLS アンダーレイ 設計理由の深掘り

## 質問

pattern2_q7.md（Task 1.7）の解答について、さらに以下を知りたい：
- ACL はどんな考え方で設定されているか？
- トランジットリンクの情報を含めないことでどんないいことがあるのか
- なぜ autoconfig を使っているのか
- R1 において `ip ospf 1 area 0` を Gi0/2 にしている理由は何か。Gi0/1 には OSPF はいらないのか
- PE ルーターにおいて `mpls ldp password required for` がいらないのはなぜか

## 回答

### ① ACL（access-list 11）の考え方
`access-list 11` は**パケットフィルタではなく、LDP パスワードを適用するピアの集合（グループ）を定義するリスト**。標準 ACL でマッチさせる対象は **LDP ピアのルータ ID（＝各ルータの Loopback0 アドレス）**。

- R1 → `permit .2/.3/.5`：直接 LDP を張る R2・R3・R5 の Loopback を列挙
- R2 → `permit .1/.4/.6`：R1・R4・R6

考え方は「このルータが認証したい直接 LDP ネイバーの LDP-ID を全部並べる」。`mpls ldp password option 1 for 11` の `for 11` でこの集合に一括適用するため、ピアが多いハブ（R1/R2）でも1つの ACL でまとめてポリシーを当てられる（＝グループ/グローバル設定の実体）。

### ② トランジットリンク情報を LSA に含めないメリット
要件充足に加えて実運用上の利点：
- **LSDB / RIB の縮小**：コア間リンク（/30 等）が全ルータに載らない → メモリ・SPF 負荷を削減
- **インフラ隠蔽（セキュリティ）**：コアのリンクアドレスが OSPF ドメインや再配布先に漏れない → 攻撃対象化しにくい
- **MPLS では中継リンクの IP 到達性が不要**：転送はラベル、次ホップは Loopback（LDP/BGP）で解決 → 配っても使われないものを配らないのが合理的
- **収束・安定性**：不要プレフィックス減でフラップ時の再計算・再広告が減る

### ③ なぜ autoconfig を使うのか
`mpls ldp autoconfig area 0` は OSPF area 0 の全 IF で LDP を自動有効化する。
- **設定漏れ防止・一貫性**：各コアリンクへの `mpls ip` 打ち忘れを防ぎ、「IGP が通る範囲 = ラベルが通る範囲」を自動一致
- **IGP と LDP の整合**：ラベル未張り区間による MPLS ブラックホールを防ぐ
- **運用性**：新コアリンクを OSPF に足せば自動で LDP も有効
- 補足：本解答は保険で各 IF に `mpls ip` も明示。厳密には autoconfig があれば `mpls ip` は必須でない

### ④ R1 の Gi0/2 に OSPF、Gi0/1 はいらないのか
**結論：Gi0/1 にも OSPF は入っている。** config は `interface range GigabitEthernet0/0-2` で、**Gi0/0・Gi0/1・Gi0/2 の3本すべて**に `ip ospf 1 area 0` / `ip ospf network point-to-point` / `mpls ip` を一括適用する range コマンド（Gi0/2 だけではない）。

配線（EI_v2.yaml links）：
| R1 の IF | 対向 | 行 |
|---|---|---|
| Gi0/0 | R2 Gi0/0 | 4651 |
| Gi0/1 | R3 Gi8 | 4658 |
| Gi0/2 | R5 Gi8 | 4686 |

3本ともコア内リンク → 全てで OSPF ネイバーが必要（要件「すべての内部リンクで OSPF ネイバー確立」）。個別に書くなら3本に同じ3行を書くことになり、それを1行で済ませているのが range。

### ⑤ PE で `mpls ldp password required for` がいらない理由
- R1/R2 は2段構え：`password option 1 for 11 …`（グループにパスワードを用意）＋ `password required for 11`（必須化）。option だけだと optional（パスワードが無いピアとも繋がりうる）ため、要件「すべての LDP ピアにパスワード必須」を満たすには `required` で未認証セッションを明示禁止する必要がある。
- PE（R3〜R6）は `mpls ldp neighbor <ip> password …` で特定1ピアに直接パスワードを指定。TCP MD5 は両端一致が前提で、鍵が無ければセッション自体が成立しない ＝ そのピアには「必須」がコマンドに内包されている。
- 要件面でも PE は「ピア単位認証」のみで「全ピア必須化」は R1/R2 だけ。PE は ACL グループが無いので `required for <acl>` を使う場面がない。
- まとめ：`required` は「グループ内の（未指定ピアも含めた）強制」のためのもの。PE は個別ピア指定なので不要。

## 補足Q: `mpls ldp router-id` / `neighbor password` は autoconfig で OSPF に「適用」される？

**いいえ。そこは別々のレイヤで、autoconfig が router-id や password を OSPF に適用しているわけではない。**

3つのコマンドは役割（レイヤ）が違う：

| コマンド | レイヤ | 役割 |
|---|---|---|
| `mpls ldp autoconfig area 0`（router ospf 配下） | **インターフェース単位** | OSPF area 0 の全 IF で LDP を自動有効化。「**どこで**LDP を動かすか」を OSPF の IF リストに合わせる（`mpls ip` 相当の自動化） |
| `mpls ldp router-id Loopback0 force`（グローバル） | **LDP プロセス全体** | LDP の識別子（router-id）を Loopback0 に固定。IF や OSPF とは独立 |
| `mpls ldp neighbor 100.255.254.2 password …`（グローバル） | **LDP セッション（TCP）** | 特定ピアとの TCP セッションを MD5 認証。IF ではなくセッションのパラメータ |

つまり autoconfig は「**LDP を有効にするインターフェースの範囲**」を決めるだけ。router-id と password は autoconfig とは無関係に LDP プロセス／セッションへ効く（autoconfig が"適用"する対象ではない）。

### 実際の連携（LDP セッション確立の流れ）
独立した設定だが、セッション確立の順序で協調して働く：

```
① mpls ldp autoconfig area 0
   → OSPF area 0 の各 IF で LDP 有効化 → その IF から LDP hello (UDP 646) を送出
② リンク上でネイバーを発見
③ 双方の LDP router-id（= Loopback0, mpls ldp router-id で設定）間で
   TCP セッション (TCP 646) を確立
   ※ Loopback0 は OSPF area 0 で広告済みなので到達可能
④ その TCP セッションを neighbor password の MD5 で認証
```

- router-id と password は ③④ の「セッション確立フェーズ」で効く。
- autoconfig は ① の「インターフェース有効化フェーズ」を担うだけで、router-id / password を"適用"はしない。
- 補足：LDP の router-id と OSPF の router-id は別物（たまたま両方 Loopback0 でも独立した設定）。

### 正しい理解
- ×「router-id / password の LDP 設定を autoconfig で OSPF に適用している」
- ○「autoconfig は OSPF area 0 の IF に **LDP を有効化**するだけ。その上で LDP は Loopback0 の router-id 同士で TCP セッションを張り、それを password で認証する。3者はレイヤの違う別設定」
