Template の仕組みと関係性

SD-WAN Template 階層の考え方と、各タスクで作る/編集するテンプレートの意味を理解する

📐 Template の基本構造

SD-WAN の Template は 2層構造：

Device Template (DT)
＝
Feature Template (FT)
×
複数個の組み合わせ
  • Device Template (DT) = デバイス1台に対応する「設定の入れ物」。実際にデバイスに Attach するのはこれ
  • Feature Template (FT) = VPN設定・Interface設定・OSPF設定など、個々の機能を定義するパーツ
  • DT の中に複数の FT を組み込んで、1台分の完全な設定を構成する
  • Device Specific 変数 = 同じ DT を複数台に使い回す時、デバイスごとに異なる値（IP、Hostname等）を Attach 時に入力

♻️ 共有 vs 専用 テンプレート

FT は 複数の DT から参照できる（共有）= 変更が全デバイスに波及

  • 共有 cEdge System Template — 全 cEdge で使用。System IP/Hostname は Device Specific
  • 共有 cEdge-AAA-Template — 認証設定。全デバイス共通
  • 共有 cEdge-CLI-addon — CLI アドオン。全デバイス共通
  • 共有 OSPF-BR-cEdge-Template — Branch OSPF 設定。cEdge40/51/52 が共有
  • 共有 cEdge-VPN999-Inf_Template — VPN999 Interface。DC・Branch 共通
  • 専用 CCIE-DC-OSPF-VPN0-Template — DC の VPN0 OSPF のみ (Task 2.1)
  • 専用 CCIE-BR1-VPN999-Template — cEdge40 専用の VPN999 (Task 2.2)
  • 専用 CCIE-BR2-* 系 — Branch #2 専用 (Task 2.5)
💡 共有 FT を編集すると、それを使う全 DT に影響する。影響範囲を常に意識すること！

🔗 DT → FT の組み込み構造

Device Template にはスロットがある：

  • System — System Template + AAA Template
  • Transport & Management VPN (VPN 0) — VPN Template + Interface Template(s) + OSPF/BGP Template
  • VPN 512 — Management VPN + Interface
  • Service VPN (VPN 999) — VPN Template + Interface Template(s) + OSPF Template
  • Additional Templates — CLI Add-On, Localized Policy など

各スロットに FT を選択して埋める → 全スロットが埋まったら Attach 可能

⚠️ 「Global」 vs 「Device Specific」 vs 「Default」

FT 内の各パラメータの設定方法：

  • Global — 全デバイスで同じ固定値（例: OSPF Area 0）
  • Device Specific — Attach 時にデバイスごとに入力（例: IP アドレス、Hostname）
    → 試験で暗記が必要な値はこれ！
  • Default — Cisco デフォルト値をそのまま使用
💡 IP MTU を Device Specific にする理由: cEdge21 と cEdge22 で同じ FT を使いつつ、MTU 値だけ個別指定できるから

OSPF は Branch 内の通信だけでなく、学習した経路を OMP に注入して全サイトに配布する入口。逆方向も同じ。

OMP vs BGP — 完全に役割が違う

  • OMP — サイト間の 全 経路交換 (VPN999, TLOC, Policy)。Branch 間も OMP。BGP ではない
  • BGP — overlay の外側だけ: ① DC Underlay (R22↔cEdge, AS 65002) ② TLOC Extension (cEdge51/52↔ISP, AS 10001 戻り経路広報)

Color = WAN 回線のラベル

private1 private2 は物理 WAN 回線 (SP1/SP2) を識別するタグ。同じ color 同士で IPsec トンネル (TLOC) を形成。

  • SP ルータの設定は不要 — cEdge の WAN IF に color を貼るだけ
  • AAR の Preferred Color で特定の WAN にトラフィックを誘導
  • DSCP 値は経路選択に影響しない — BFD probe の品質測定用マーカー

v1.3 は Branch 専用 VPN999 Template を作る

v1.1 は DC/Branch で VPN-999-Template を共有 → Parent IF Template が必要だった。

v1.3 は Branch 専用の VPN999 FT を新規作成するため、Parent IF 不要・Variable List もシンプル。

CCIE-BR1-VPN999-Template (cEdge40 専用) / CCIE-BR2-VPN999-Template (cEdge51/52 共有)

TLOC Extension の仕組み — なぜ BGP が必要か

L2 延長: cEdge51 の Gi3 と cEdge52 の Gi3 を 802.1Q VLAN で L2 接続。互いの WAN 回線を「借用」して 2色 TLOC を実現。

  • cEdge51: 自前=private1 (Gi2) / 借用=private2 (Gi3.3521 from cEdge52)
  • cEdge52: 自前=private1 (Gi2) / 借用=private2 (Gi3.3512 from cEdge51)
  • Tunnel Group で DC 側の接続先を固定 (100→cEdge21, 200→cEdge22)

BGP の役割: ISP (AS 10001) に対して 戻り経路を広報するためだけに使用。

  • ISP が Branch #2 宛のパケットを正しい WAN に返すために必要
  • SD-WAN overlay 内の経路交換には一切関係ない (それは OMP)
  • CCIE-BR2-BGP-VPN0-Template で neighbor/AS を定義

各タスクで何を作る/編集するか & その理由

Task 2.1 — DC Underlay: なぜ Template 拡張か

やること: DC cEdge21/22 の VPN0 に OSPF を追加、VPN999 Interface に MTU 設定

  • NEW CCIE-DC-OSPF-VPN0-Template — VPN0 用の OSPF FT を新規作成
  • EDIT cEdge-VPN0-Inf_Template — OSPF 有効化 (Global → ON)
  • EDIT cEdge-VPN999-Inf_Template — IP MTU を Device Specific に変更
  • EDIT DC-cEdge_Device_Template — 上記 OSPF FT を VPN0 に追加

💡 CLI 直打ちではなく Template 拡張にする理由: vManage が唯一の設定管理元（Authoritative Source）を維持できるから。CLI で変更すると out-of-band 扱いで監査不能になる

R22 のみ CLI 管理対象（vManage Template 管理外）→ BGP/OSPF/route-map は直接設定

Task 2.2 — Branch #1: なぜ新規 DT か

やること: cEdge40 用の Device Template を新規作成し Attach

  • NEW CCIE-BR1-VPN999-Template — VPN999 + OMP Aggregate 設定（集約広報）
  • NEW BR1-CCIEDGE40-Device-Template — FT を組み合わせた DT

共有 FT: System, AAA, VPN0, VPN512, CLI-addon はそのまま再利用 → Branch 固有の部分だけ新規作成

💡 OMP Aggregate は VPN999-Template 内で設定。Prefix = 10.4.0.0/16 を Device Specific で指定し、Aggregate Only = ON にすることで、/30 の詳細ルートではなく /16 集約のみ OMP 広報

Task 2.3 — Hub&Spoke: vSmart Template + Central Policy

やること: vSmart 用 DT を作成し、Central Policy で Branch 間直接通信を禁止

Task 2.4 — AAR: Central Policy に Traffic Rules 追加

やること: 2.3 で作った CCIE-CentralPolicy に AAR ルールを追加

  • CCIE-Queue → Class Map（Localized Policy Lists で作成）
  • CCIE-SSH → Application List（Secure Shell）
  • CCIE-Probe → App Probe Class（private1: DSCP 18/AF21, private2: DSCP 28/AF32）
  • CCIE-SLA-Traffic → SLA Class（Loss < 4%）
  • CCIE-AAR → App Route Policy（Match SSH → SLA → preferred private1）

💡 AAR は BFD probe で各色の品質を測定 → SLA 条件を満たす色を自動選択。Central Policy の Traffic Rules セクションに追加
