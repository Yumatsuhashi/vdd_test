1. MST（Task 1.3）

確定情報（実問題文）: P4 Task 1.3 は MST によるL2ループ防止 + トラフィックエンジニアリング。PAgP/EtherChannelの記述は なし。
実際の英語問題文
"Implement a scalable Spanning Tree solution in the HQ site that minimizes internal switch CPU resources by grouping VLANs.
• Use "CCIE" as the identification string for the region with a revision level of 1.
• Ensure that traffic for VLAN 2000 is natively forwarded through SW101 as the primary path, while SW102 acts as the immediate standby.
• Inversely, ensure VLAN 2001 traffic prefers SW102 as its primary forwarding path, with SW101 serving as the backup.
• SW110 must never become the root bridge for any instance under normal operating conditions."
読み解き:
・"scalable Spanning Tree" + "minimizes CPU by grouping VLANs" → MST（PVST+ はVLAN毎にSTPインスタンス=CPU重い。MSTはグルーピングで軽減）
・"identification string" + "region" → spanning-tree mst configuration → name CCIE
・"revision level of 1" → revision 1
・"VLAN 2000 → SW101 primary, SW102 standby" → Instance 1 に VLAN 2000 をマッピング、SW101 = root
・"VLAN 2001 → SW102 primary, SW101 backup" → Instance 2 に VLAN 2001 をマッピング、SW102 = root
・"SW110 must never become root" → SW110 は高 priority（明示的に 61440 等に設定）

3. OSPF Multi-Area（Task 1.5）

確定情報: P4ではOSPF multi-area設定が新要件で出る。
予想英語問題文（遠回し表現）
"The network between R21, R22, and the distribution switches (SW101, SW102) should be isolated from external routing information originating outside the autonomous system. These switches do not require visibility into routes learned from BGP redistribution — a default route from the border routers is sufficient for reaching external destinations.

Ensure OSPF adjacency is maintained between R21/R22 and the switches while implementing this design change."
読み解き:
・"isolated from external routing information" → Type 5 LSA ブロック → Stub Area
・"do not require visibility into routes learned from BGP redistribution" → 外部ルート不要
・"default route from the border routers is sufficient" → ABRがdefault生成 = stubの動作
・"Ensure OSPF adjacency is maintained" → 全台で area X stub を一致させることのヒント
→ 答え: SW群のIFを Area 1 に移し、area 1 stub を R21/R22/SW101/SW102 全台に設定
予想英語問題文（Totally Stubby パターン）
"Furthermore, the distribution switches should only maintain a single default route for all inter-area and external destinations. Minimize the LSDB size on these switches as much as possible."
読み解き:
・"single default route for all inter-area AND external" → Type 3 も Type 5 も不要 = Totally Stub
・"Minimize LSDB" → no-summary
→ ABR (R21/R22) のみ: area 1 stub no-summary、SW101/SW102: area 1 stub（no-summaryは付けない）

4. iBGP R61-R62（Task 1.6）

確定情報: P4のTask 1.6はEIGRPではなくiBGP（R61-R62間）に変更。
予想英語問題文（遠回し表現）
"R61 and R62 belong to the same administrative domain. Establish a routing peering between them using their Loopback0 addresses. Ensure that prefixes learned from external peers can be properly forwarded between R61 and R62 without requiring a third-party route reflector."
読み解き:
・"same administrative domain" → 同AS → iBGP（remote-as 65006）
・"Loopback0 addresses" → update-source Loopback0
・"prefixes learned from external peers can be properly forwarded" → next-hop-self 必須
・"without requiring a third-party route reflector" → 2台だけなのRR不要だがnext-hop-selfは必要
iBGP 基本設定

  neighbor 10.6.255.62 activate
  neighbor 10.6.255.62 next-hop-self   ! eBGPから学んだルートの転送に必要

R62:
router bgp 65006
 neighbor 10.6.255.61 remote-as 65006
 neighbor 10.6.255.61 update-source Loopback0
 !
 address-family ipv4 unicast
  neighbor 10.6.255.61 activate
  neighbor 10.6.255.61 next-hop-self

RR データプレーン解説

⚠ IP SLA + Track の核心ルール
EEM event track は「状態変化」でしか発火しない:
・ずっと UP → 発火しない
・ずっと DOWN → 発火しない
・UP → DOWN（到達不能になった）→ state down / state any で発火
・DOWN → UP（復旧した）→ state up / state any で発火

だから:
・ping 先を 自分自身にしてはダメ（常に成功 → 永遠に状態変化なし）
・ping 先は 別ルーター（例: R24 の 10.2.255.24）にする

"On R23, configure reachability monitoring toward R24 (10.2.255.24) using ICMP probes every 10 seconds. When the reachability state changes, R23 should automatically collect interface status, routing table, and probe statistics via a GuestShell Python script, appending the results with a timestamp to /bootflash/guest-share/ping_report.txt. Generate a syslog notification upon completion."
読み解き:
・"reachability monitoring" + "ICMP probes every 10 seconds" → ip sla 1 + icmp-echo + frequency 10
・"reachability state changes" → track 1 ip sla 1 reachability + EEM event track 1 state any
・"automatically collect ... via GuestShell Python script" → EEM → guestshell run python3
・"appending the results with a timestamp" → Python: datetime.now() + open("a+")
・"syslog notification" → action 3.0 syslog priority notifications msg "..."
