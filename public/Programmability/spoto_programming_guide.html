3.1

EEM × GuestShell × Python 自動診断

🎯 タスク概要

ルーター R23 上に Loopback10 を作成し、EEM アプレットと GuestShell を連携させて、Loopback10 のダウン検知時に自動で RCA（Root Cause Analysis）診断データを収集・ファイル保存する仕組みを構築する。

解決する課題：
・障害発生時にエンジニアが手動でコマンドを叩く必要がある → 自動化
・証跡データの収集漏れを防ぐ → ファイル保存
・復旧後に原因調査できるようにする → RCA データの蓄積

⛓️ 動作フロー

① Loopback10 がダウン 障害のシミュレーション対象。shutdown で意図的に発生させる。
② Object Track 1 が状態変化を検知 track 1 interface Loopback10 line-protocol — プロトコル状態を常時監視し、down を検出する。
③ EEM アプレット "RCA" が起動 event track 1 state down ratelimit 5 — Track 1 が down になった時に発火。ratelimit 5 でチャタリング防止（5秒に1回のみ実行）。
④ GuestShell 経由で Python スクリプトを実行 action 2.0 cli command "guestshell run python3 eem_script.py loop10" — IOS-XE CLI を Python から呼び出し、結果を収集する。
⑤ 診断データをファイルに保存 & syslog 記録 4つの show コマンド出力を bootflash:/guest-share/RCA_Capture.txt に追記。syslog に完了メッセージを出力。

① Configure OSPF and Loopback10

まず configure terminal に入り、ロギングを設定したうえで、監視対象インターフェース（Loopback10）、GuestShell 接続用インターフェース（VPG0）、OSPF、Object Tracking を構成する。

完全コンフィグ 
configure terminal
logging console

! ── Loopback10（監視対象）──
interface Loopback10
 ip address 192.168.1.1 255.255.255.0

! ── VirtualPortGroup0（GuestShell との L3 接続）──
interface VirtualPortGroup0
 ip address 192.168.0.1 255.255.255.0

! ── OSPF ──
router ospf 1
 network 192.168.0.0 0.0.0.255 area 0
 passive-interface VirtualPortGroup0

! ── Object Tracking ──
track 1 interface Loopback10 line-protocol

コンフィグ行ごとの解説

コマンド解説
configure terminal グローバルコンフィグモードに移行。以降のすべての設定はこのモード内で実行する。
logging console コンソールへの syslog 出力を有効化。EEM の action 3.0 で送信される syslog メッセージ（"RCA Data Capture Complete"）をリアルタイムで確認できるようにする。
interface Loopback10 RCA 診断のトリガーとなる監視対象インターフェースを作成。Loopback は物理的に存在しないため、shutdown で意図的に障害をシミュレーションできる。
ip address 192.168.1.1 255.255.255.0 Loopback10 に IP アドレスを付与。これにより line-protocol が up になり、Object Track の初期状態が確立される。
interface VirtualPortGroup0 GuestShell コンテナとルーター間の仮想 L3 インターフェース。GuestShell が外部と通信するための唯一の経路。
ip address 192.168.0.1 255.255.255.0 ルーター側の IP。この IP が GuestShell のデフォルトゲートウェイになる（192.168.0.1 → 192.168.0.2 の通信パス）。
router ospf 1 OSPF プロセス 1 を起動。GuestShell サブネットを他のルーターに広告するために使用。
network 192.168.0.0 0.0.0.255 area 0 VPG0 のサブネット（192.168.0.0/24）を OSPF area 0 に広告。GuestShell が外部ネットワークと通信可能になる。
passive-interface VirtualPortGroup0 VPG0 で OSPF Hello パケットの送受信を抑止。GuestShell は OSPF ルーターではないため、ネゴシエーションは不要。ネットワーク広告だけを行う。
track 1 interface Loopback10 line-protocol Loopback10 の line-protocol 状態（up/down）を常時監視するオブジェクトを作成。この Track オブジェクトが EEM のトリガー条件になる。

② Configure EEM

Object Track 1 が down を検知した時に自動実行される EEM アプレットを定義する。

完全コンフィグ 
event manager applet RCA
 event track 1 state down ratelimit 5
 action 1.0 cli command "enable"
 action 2.0 cli command "guestshell run python3 eem_script.py loop10"
 action 3.0 syslog priority notifications msg "RCA Data Capture Complete"

各行の解説

コマンド解説
event manager applet RCA EEM アプレット "RCA" を定義。アプレットはイベント駆動型の自動化スクリプトで、指定条件が満たされた時に一連の action を順次実行する。
event track 1 state down ratelimit 5 トリガー条件：Track 1（= Loopback10 の line-protocol）が down 状態になった時にアプレットを起動する。

ratelimit 5 → 最低5秒の間隔を空けないと再実行されない。インターフェースが頻繁にフラップ（up/down 繰り返し）した場合のチャタリング対策。

action 解説（実行順序）

Actionコマンド解説
1.0 cli command "enable" 特権 EXEC モードに移行。EEM アプレットはデフォルトで User EXEC モードで実行される。後続の guestshell run コマンドには特権が必要なため、最初に enable する必要がある。
2.0 cli command "guestshell run python3 eem_script.py loop10" GuestShell 内で Python スクリプトを実行。
・guestshell run → IOS-XE CLI から GuestShell 内のコマンドを呼び出すプレフィックス
・python3 eem_script.py → GuestShell 内の Python3 インタプリタでスクリプトを実行
・loop10 → スクリプトに渡される引数（sys.argv[1] で受け取り、"show interface loop10" として使用される）
3.0 syslog priority notifications msg "RCA Data Capture Complete" syslog にメッセージを記録。
・priority notifications → syslog severity level 5（通知レベル）
・データ収集完了の証跡として記録。logging console が有効なら、コンソールにもリアルタイム表示される。
・NMS（ネットワーク管理システム）やSIEM で RCA 実行の監査ログとしても活用可能。
action 番号の意味：
1.0, 2.0, 3.0 は実行順序を決定する。EEM は番号の小さい順に action を実行する。小数を使うことで、後から間に action を挿入可能（例: 1.5 を追加して 1.0 と 2.0 の間に処理を挟める）。

③ Configure GuestShell Gateway

IOx フレームワークを有効化し、GuestShell コンテナの NIC とゲートウェイを設定する。

完全コンフィグ 
iox

app-hosting appid guestshell
 app-vnic gateway0 virtualportgroup 0 guest-interface 0
  guest-ipaddress 192.168.0.2 netmask 255.255.255.0
  exit
 app-default-gateway 192.168.0.1 guest-interface 0

各行の解説

コマンド解説
iox IOx フレームワークの有効化。GuestShell を含むすべてのアプリケーションホスティング機能の前提条件。これを入れないと GuestShell は起動できない。IOx は IOS-XE 上のコンテナ実行基盤。
app-hosting appid guestshell GuestShell アプリケーションの定義。appid guestshell は Cisco が事前定義した Linux コンテナ環境の識別子。このサブモードで NIC やゲートウェイの設定を行う。
app-vnic gateway0 virtualportgroup 0 guest-interface 0 GuestShell の仮想 NIC を VPG0 にマッピング。
・gateway0 → GuestShell 側のゲートウェイインターフェース番号
・virtualportgroup 0 → ルーター側の VPG0 に接続
・guest-interface 0 → GuestShell 内の eth0 に相当
これにより GuestShell の仮想 NIC がルーターの VPG0 と L3 で接続される。
guest-ipaddress 192.168.0.2 netmask 255.255.255.0 GuestShell 側の IP アドレスを設定。ルーター側の VPG0（192.168.0.1）と同一サブネット内に配置。これが GuestShell のネットワークアイデンティティになる。
exit app-vnic サブモードから抜ける。app-default-gateway は app-vnic の外（app-hosting 直下）で設定する。
app-default-gateway 192.168.0.1 guest-interface 0 GuestShell のデフォルトゲートウェイを設定。192.168.0.1（= ルーターの VPG0）を指定。app-hosting appid guestshell 直下のサブコマンド。GuestShell からの全通信はこのゲートウェイ経由でルーターを通過する。
GuestShell ネットワーク構成の全体像：

GuestShell (192.168.0.2) ←→ VPG0 (192.168.0.1) ←→ ルーター → OSPF → 外部ネットワーク

・GuestShell は隔離された Linux コンテナ。VPG0 が唯一の外部通信パス。
・cli モジュール経由の IOS-XE CLI 実行はこのネットワークパスとは別の内部チャネルを使用。
・OSPF で 192.168.0.0/24 を広告しているため、他のルーターから GuestShell に到達可能。

🐍 Solution — Python スクリプト（eem_script.py）完全版

#!/usr/bin/env python3
import cli
import sys

cmds = [
    "show ip interface brief",
    "show ip ospf neighbor",
    "show ip route",
    f"show interface {sys.argv[1]}",
    "show log"
]

with open("/bootflash/guest-share/RCA_Capture.txt", "a+") as f:
    for cmd in cmds:
        output = cli.execute(cmd)
        f.write(output)
スクリプト各行の解説：

import cli → Cisco IOS-XE 専用モジュール。GuestShell 内から IOS-XE CLI コマンドを実行可能にする。

import sys → コマンドライン引数を受け取るための標準モジュール。

sys.argv[1] → EEM の action 2.0 から渡される引数。ここでは "loop10" が渡され、"show interface loop10" として展開される。

open(..., "a+") → 追記モード。既存の診断データを消さずに末尾に追記する。障害が複数回発生しても履歴が蓄積される。

cli.execute(cmd) → IOS-XE CLI を実行し、結果を文字列で返す。cli.cli() と異なりヘッダー行を含まないクリーンな出力を返す。

保存先：/bootflash/guest-share/RCA_Capture.txt
→ guest-share ディレクトリは GuestShell と IOS-XE の共有領域。

収集する5つのコマンドとその目的

コマンド収集目的
show ip interface brief全インターフェースの IP アドレスと up/down 状態を概観。障害時にどのインターフェースが影響を受けたか一目で確認。
show ip ospf neighborOSPF 隣接関係（ネイバー）の状態を確認。障害がルーティングプロトコルに波及しているかを診断。
show ip routeルーティングテーブルのスナップショット。障害前後でルートがどう変化したかを後から比較可能。
show interface Loopback10障害対象インターフェースの詳細統計（パケットカウンタ、エラー、リセット回数など）。根本原因の手がかりになる。
show logsyslog バッファの内容を収集。障害発生前後のイベントログ・エラーメッセージを時系列で確認可能。

3.2

RESTCONF × Python デバッグ

🎯 タスク概要

ルーター R30 に対して RESTCONF を使い、物理インターフェースの帯域使用データ（in/out オクテット数）をプログラム的に取得する Python スクリプトのデバッグと修正を行う。

解決する課題：
・SNMP より柔軟・モダンなデータ取得方法（YANG ベースの REST API）
・特定インターフェースの in/out オクテット数を自動収集・可視化

タスクの本質：与えられたスクリプトに含まれる 4 つのバグを特定し、修正すること。

🐛 Solution — 4つのバグと修正

BUG #1

import requests が欠落

修正前（誤）import json import urllib3 ...
修正後（正）import json import requests import urllib3 ...
問題：requests モジュールが import されていないため、requests.request() 呼び出しで NameError になる。
修正：import requests を追加する。
BUG #2

接続先ホスト IP が未設定

修正前（誤）host = ""
修正後（正）host = "10.3.11.1"
問題：空文字のため、API リクエスト URL が https://:443/restconf/... となり接続先が不明。
修正：ルーター R30 の管理 IP アドレス 10.3.11.1 を設定する。
BUG #3

不要な data=payload パラメータ

修正前（誤）requests.request('get', url, auth=(...), data=payload, headers=headers, verify=False)
修正後（正）requests.request('get', url, auth=(...), headers=headers, verify=False)
問題：data=payload が渡されているが、payload 変数は未定義（NameError）。そもそも GET リクエストに request body は不要。
修正：data=payload の行を完全に削除する。
BUG #4

response.json に括弧 () が欠落

修正前（誤）jsondata = response.json
修正後（正）jsondata = response.json()
問題：response.json はメソッドオブジェクトへの参照に過ぎない。() を付けないと JSON パースが実行されず、jsondata にはメソッド自体が代入される。後続のキーアクセスで TypeError になる。
修正：response.json() と括弧を付けてメソッドを呼び出す。

✅ Solution — 修正済み完全スクリプト

# ============================================
# RESTCONF Interface Statistics Collector
# Target: Router R30 (10.3.11.1)
# ============================================

import json
import requests                             # ← Bug #1 修正: import requests を追加
import urllib3

urllib3.disable_warnings(urllib3.exceptions.InsecureRequestWarning)

host = "10.3.11.1"                          # ← Bug #2 修正: IP アドレスを設定
port = "443"
url = f"https://{host}:{port}/restconf/data/Cisco-IOS-XE-interfaces-oper:interfaces/interface"

headers = {
    "Accept": "application/yang-data+json",
    "Content-Type": "application/yang-data+json"
}

response = requests.request(
    'get',
    url,
    auth=("netadmin", "CCIEinfra"),
                                            # ← Bug #3 修正: data=payload を削除
    headers=headers,
    verify=False
)

jsondata = response.json()                  # ← Bug #4 修正: () を追加

print("-" * 100)
print(f"{'Interface':<45} {'Ingress Octets':<30} {'Egress Octets':<30}")
print("-" * 100)

for interface in jsondata["Cisco-IOS-XE-interfaces-oper:interface"]:
    if (interface['oper-status'] == 'if-oper-state-ready' and
        interface['interface-type'] == 'iana-iftype-ethernet-csmacd'):

        switchport = interface["name"]
        ingressoctets = interface["statistics"]["in-octets"]
        egressoctets = interface["statistics"]["out-octets"]
        print(f"{switchport:<45} {ingressoctets:<30} {egressoctets:<30}")

3.3

DNA Center API エンドポイント特定

🎯 タスク概要

提示された JSON レスポンスを分析し、それがどの DNA Center（DNAC）API エンドポイントから返されたものかを論理的に特定する推論問題。

📄 提示された JSON レスポンス

{ "response": { "managementIpAddr": "10.4.255.400", "haStatus": "Non-redundant", "serialNumber": "...", "reachabilityStatus": "UNREACHABLE", "hostname": "sw400.ccie.cisco.com", "platformId": "c9300-24T", "nwDeviceid": "...", "deviceRole": "ACCESS", "deviceFamily": "Switches and Hubs", "macAddress": "...", "collectionStatus": "", "deviceSeries": "Cisco Catalyst 9300 Series Switches", "overallHealth": -2, // ← 決定的フィールド "memoryScore": -1, // ← 決定的フィールド "cpuScore": -1, // ← 決定的フィールド "location": "Global/Branch1-Bldg1", "timestamp": 1675353000000 // ← 決定的フィールド } }

🧠 Solution — 論理的な判定プロセス

STEP 1

決定的なフィールドを特定する

JSON 内に overallHealth、memoryScore、cpuScore、timestamp が含まれている。これらは全てのエンドポイントで返されるわけではなく、候補の絞り込みに使える。

STEP 2

各エンドポイントとフィールドの照合

JSON フィールド /network-device /device-enrichment-details /device-details
managementIpAddr ✔✔✔
serialNumber ✔✔✔
reachabilityStatus ✔✔✔
overallHealth ✘✔✔
memoryScore ✘✘✔
cpuScore ✘✘✔
location ✘✔✔
timestamp ✘✘✔

ハイライトされた行が決定的なフィールド。memoryScore, cpuScore, timestamp の3つが揃うエンドポイントは /device-details のみ。

STEP 3

消去法による各候補の除外理由

/dna/intent/api/v1/network-device ではない理由：
・overallHealth, memoryScore, cpuScore, location, timestamp がいずれも含まれない
・このエンドポイントはデバイスの基本情報（hostname, serialNumber, reachabilityStatus 等）のみを返す
/dna/intent/api/v1/device-enrichment-details ではない理由：
・memoryScore, cpuScore, timestamp が含まれない
・リクエスト形式が異なる：ヘッダーに MAC アドレスや IP を渡す方式
・レスポンス構造が異なる：deviceDetails / issueDetails のネスト構造であり、提示された JSON のようなフラット構造ではない
正解

/dna/intent/api/v1/device-details

根拠①：memoryScore と cpuScore は device-details エンドポイント固有のフィールド。他のエンドポイントでは返されない。

根拠②：timestamp（エポックミリ秒）の存在は、特定時点のスナップショットデータであることを示す。device-details の特徴。

根拠③：overallHealth（健全性スコア）も含まれている。健全性情報を含む詳細エンドポイントの特徴。

根拠④：レスポンスがフラットな JSON 構造。device-enrichment-details のようなネスト構造ではない。
